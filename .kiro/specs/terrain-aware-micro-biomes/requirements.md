# Requirements Document

## Introduction

The current micro-biome system in the procedural world generation engine has two critical defects that prevent it from functioning as designed. First, the frequency parameter calculation is mathematically incorrect: it assumes noise values in the range [0, 1] when the actual fBM noise returns values in [-1, 1], causing the threshold formula to produce incorrect results where maximum frequency (0.5) yields almost no micro-biomes. Second, while terrain-aware placement logic exists for depression-type (OASIS, POND) and flat-terrain-type (CLEARING, GROVE) micro-biomes, the placement conditions may be too strict or the visual representation insufficient, making micro-biomes difficult to observe even when correctly placed. This feature corrects the frequency parameter bug, improves terrain-aware placement algorithms, adds visual feedback mechanisms, and provides comprehensive property-based tests to validate correct behaviour across the full parameter range.

## Glossary

- **EnhancedBiomeSystem**: The class in `src/world/enhanced-biome.ts` responsible for biome classification with transitions, micro-biomes, and elevation bands.
- **MicroBiomeType**: An enumeration of micro-biome variants: OASIS (desert), CLEARING (forest), POND (plains), GROVE (tundra).
- **Micro-Biome Frequency**: A configuration parameter (range 0.0–0.5) controlling the density of micro-biome placement; 0.0 means no micro-biomes, 0.5 means maximum density.
- **fBM Noise**: Fractional Brownian Motion noise generated by `NoiseEngine.fbm`, returning values in the range [-1, 1].
- **Noise Threshold**: The computed value used to determine whether a noise sample qualifies for micro-biome placement; derived from the frequency parameter.
- **Terrain Depression**: A local terrain minimum identified by comparing a point's height to the average height of its four cardinal neighbours; a depression exists when `(height - neighbourAvg) < -depressionDepthThreshold`.
- **Terrain Gradient**: The root-mean-square magnitude of height differences between a point and its four cardinal neighbours; low gradient indicates flat terrain.
- **Depression Depth Threshold**: A configuration parameter (default 0.05) defining the minimum depth below neighbourhood average required for depression-type micro-biome placement.
- **Clearing Gradient Threshold**: A configuration parameter (default 0.03) defining the maximum terrain gradient allowed for flat-terrain-type micro-biome placement.
- **Parent Biome**: The primary biome type (DESERT, FOREST, PLAINS, TUNDRA) within which a micro-biome may be placed.
- **Visual Feedback**: Mechanisms to make micro-biomes visually distinct from their parent biomes, including colour variation, debug overlays, or statistical displays.
- **Property-Based Test**: A test that validates a property holds across a large generated input space using a library like fast-check.
- **Demo Application**: The interactive browser-based application in `demo/` that visualises generated worlds and exposes configuration controls.

## Requirements

### Requirement 1: Correct Micro-Biome Frequency Parameter Calculation

**User Story:** As a world-builder, I want the micro-biome frequency parameter to work correctly across its full range (0.0 to 0.5), so that adjusting the slider produces observable changes in micro-biome density.

#### Acceptance Criteria

1. THE EnhancedBiomeSystem SHALL compute the noise threshold for micro-biome placement using a formula that correctly accounts for fBM noise values in the range [-1, 1].
2. WHEN `microBiomeFrequency` is 0.0, THE EnhancedBiomeSystem SHALL produce a threshold that prevents all micro-biome placement (no noise value passes the threshold).
3. WHEN `microBiomeFrequency` is 0.5 (maximum), THE EnhancedBiomeSystem SHALL produce a threshold that allows maximum micro-biome density within terrain constraints.
4. FOR ALL values of `microBiomeFrequency` in the range [0.0, 0.5], THE EnhancedBiomeSystem SHALL produce monotonically increasing micro-biome density as frequency increases (higher frequency → more micro-biomes).
5. THE EnhancedBiomeSystem SHALL replace the current threshold formula `threshold = 1.0 - microBiomeFrequency * 2` with a corrected formula that maps the [-1, 1] noise range to the [0.0, 0.5] frequency range.

### Requirement 2: Improved Depression Detection for OASIS and POND Micro-Biomes

**User Story:** As a world-builder, I want oases to appear in desert depressions where water would naturally collect, so that micro-biomes feel geographically plausible.

#### Acceptance Criteria

1. WHEN evaluating a candidate position for OASIS or POND micro-biome placement, THE EnhancedBiomeSystem SHALL sample terrain height at the candidate position and at its four cardinal neighbours (±1 world unit in X and Y).
2. THE EnhancedBiomeSystem SHALL compute the neighbourhood average height as `(height(x+1,y) + height(x-1,y) + height(x,y+1) + height(x,y-1)) / 4`.
3. THE EnhancedBiomeSystem SHALL classify a position as a depression WHEN `(centerHeight - neighbourAvg) < -depressionDepthThreshold`.
4. IF a candidate position is not a depression, THEN THE EnhancedBiomeSystem SHALL not place an OASIS or POND micro-biome at that position, regardless of noise value.
5. THE EnhancedBiomeSystem SHALL expose `depressionDepthThreshold` as a configuration parameter in `EnhancedBiomeConfig` with a default value of 0.05.
6. WHEN `depressionDepthThreshold` is 0.0, THE EnhancedBiomeSystem SHALL classify all positions as potential depressions, falling back to noise-only placement logic.

---

### Requirement 3: Improved Flat-Terrain Detection for CLEARING and GROVE Micro-Biomes

**User Story:** As a world-builder, I want forest clearings to appear on flat terrain rather than on slopes, so that clearings feel like natural meadows rather than random noise artefacts.

#### Acceptance Criteria

1. WHEN evaluating a candidate position for CLEARING or GROVE micro-biome placement, THE EnhancedBiomeSystem SHALL compute the terrain gradient magnitude at that position.
2. THE EnhancedBiomeSystem SHALL compute gradient magnitude as the root-mean-square of height differences: `sqrt((dx1² + dx2² + dy1² + dy2²) / 4)`, where `dx1 = height(x+1,y) - height(x,y)`, `dx2 = height(x-1,y) - height(x,y)`, `dy1 = height(x,y+1) - height(x,y)`, `dy2 = height(x,y-1) - height(x,y)`.
3. THE EnhancedBiomeSystem SHALL classify a position as flat terrain WHEN `gradient < clearingGradientThreshold`.
4. IF a candidate position is not flat terrain, THEN THE EnhancedBiomeSystem SHALL not place a CLEARING or GROVE micro-biome at that position, regardless of noise value.
5. THE EnhancedBiomeSystem SHALL expose `clearingGradientThreshold` as a configuration parameter in `EnhancedBiomeConfig` with a default value of 0.03.
6. WHEN `clearingGradientThreshold` is set to a very high value (e.g., 1.0), THE EnhancedBiomeSystem SHALL classify all positions as flat terrain, falling back to noise-only placement logic.

### Requirement 4: Visual Distinction for Micro-Biomes

**User Story:** As a user of the demo application, I want micro-biomes to be visually distinct from their parent biomes, so that I can observe the effect of the frequency parameter and terrain-aware placement.

#### Acceptance Criteria

1. THE Demo Application SHALL render micro-biome tiles with a colour that differs from the parent biome colour by at least 20% in at least one RGB channel.
2. THE Demo Application SHALL provide a debug visualisation mode that highlights micro-biome locations with a distinct overlay colour or pattern.
3. WHEN the debug visualisation mode is enabled, THE Demo Application SHALL display a count of micro-biomes currently visible in the viewport.
4. THE StatisticsDisplay component SHALL include a "Micro-Biomes Visible" counter that updates in real-time as the camera moves.
5. THE ControlPanel component SHALL include a toggle control labelled "Highlight Micro-Biomes" that enables/disables the debug visualisation mode.

---

### Requirement 5: Observable Frequency Parameter Behaviour

**User Story:** As a user of the demo application, I want to see immediate visual feedback when adjusting the micro-biome frequency slider, so that I can verify the parameter is working correctly.

#### Acceptance Criteria

1. THE ControlPanel component SHALL include a slider control for `microBiomeFrequency` with a range of 0.0 to 0.5 and a step size of 0.01.
2. WHEN the user adjusts the micro-biome frequency slider, THE Demo Application SHALL regenerate visible chunks within 500 milliseconds.
3. WHEN the user sets `microBiomeFrequency` to 0.0, THE Demo Application SHALL display zero micro-biomes in the statistics panel.
4. WHEN the user sets `microBiomeFrequency` to 0.5 and views a region containing at least 200 biome tiles, THE Demo Application SHALL display at least 5 micro-biomes in the statistics panel (assuming terrain conditions allow placement).
5. THE ControlPanel component SHALL display the current `microBiomeFrequency` value as a numeric label next to the slider, formatted to two decimal places.

---

### Requirement 6: Determinism and Performance

**User Story:** As a library consumer, I want micro-biome generation to remain deterministic and meet the existing performance target, so that worlds are reproducible and generation stays within the <100 ms per-chunk budget.

#### Acceptance Criteria

1. THE EnhancedBiomeSystem SHALL produce identical micro-biome placement for the same world position, seed, and configuration across multiple calls.
2. THE EnhancedBiomeSystem SHALL not allocate heap objects during micro-biome placement evaluation; all intermediate values SHALL be computed using stack-local numeric variables.
3. WHEN generating a 32×32 chunk with `microBiomeFrequency` set to 0.5 and all terrain-aware checks enabled, THE ChunkManager SHALL complete generation in under 100 milliseconds on a modern desktop browser.
4. THE EnhancedBiomeSystem SHALL use the cached `microBiomeNoiseConfig` object for all fBM noise sampling to avoid per-tile object allocation.
5. THE EnhancedBiomeSystem SHALL pre-compute trigonometric values (cos, sin) for micro-biome size validation to avoid redundant Math function calls.

### Requirement 7: Comprehensive Property-Based Tests for Frequency Parameter

**User Story:** As a library maintainer, I want property-based tests that validate the frequency parameter works correctly across its full range, so that the bug cannot regress.

#### Acceptance Criteria

1. THE Test Suite SHALL include a property-based test that verifies: FOR ALL `microBiomeFrequency` values in [0.0, 0.5] and FOR ALL world positions, the computed noise threshold is mathematically valid for the [-1, 1] noise range.
2. THE Test Suite SHALL include a property-based test that verifies: WHEN `microBiomeFrequency` is 0.0, no micro-biomes are placed across a sample of at least 100 random world positions.
3. THE Test Suite SHALL include a property-based test that verifies: WHEN `microBiomeFrequency` is 0.5, micro-biomes are placed at a higher density than when `microBiomeFrequency` is 0.1, measured across at least 1000 random world positions with terrain conditions that allow placement.
4. THE Test Suite SHALL include a property-based test that verifies: FOR ALL pairs of frequency values `f1 < f2` in the range [0.0, 0.5], the density of micro-biomes at `f2` is greater than or equal to the density at `f1`.
5. THE Test Suite SHALL use fast-check to generate random world positions, height values, and frequency parameters for property validation.

---

### Requirement 8: Comprehensive Property-Based Tests for Terrain-Aware Placement

**User Story:** As a library maintainer, I want property-based tests that validate terrain-aware placement works correctly, so that micro-biomes only appear in geographically appropriate locations.

#### Acceptance Criteria

1. THE Test Suite SHALL include a property-based test that verifies: FOR ALL world positions with flat terrain (gradient = 0), OASIS and POND micro-biomes are never placed.
2. THE Test Suite SHALL include a property-based test that verifies: FOR ALL world positions with steep terrain (gradient > `clearingGradientThreshold`), CLEARING and GROVE micro-biomes are never placed.
3. THE Test Suite SHALL include a property-based test that verifies: FOR ALL world positions where `(centerHeight - neighbourAvg) >= -depressionDepthThreshold`, OASIS and POND micro-biomes are never placed.
4. THE Test Suite SHALL include a property-based test that verifies: FOR ALL world positions that satisfy terrain conditions (depression or flat terrain) and pass the noise threshold, the correct micro-biome type is assigned based on the parent biome.
5. THE Test Suite SHALL use fast-check to generate random terrain height functions (flat, sloped, depressed) and verify placement rules hold across at least 100 samples per property.

---

### Requirement 9: Backward Compatibility

**User Story:** As a library consumer, I want existing worlds generated with the current (buggy) micro-biome system to remain unchanged when I update the library, so that I can opt into the fix without breaking existing content.

#### Acceptance Criteria

1. THE EnhancedBiomeConfig SHALL include a `useLegacyMicroBiomeThreshold` boolean flag (default false) that preserves the old buggy threshold calculation for backward compatibility.
2. WHEN `useLegacyMicroBiomeThreshold` is true, THE EnhancedBiomeSystem SHALL use the formula `threshold = 1.0 - microBiomeFrequency * 2` to match the current behaviour.
3. WHEN `useLegacyMicroBiomeThreshold` is false (default), THE EnhancedBiomeSystem SHALL use the corrected threshold formula.
4. THE EnhancedBiomeSystem SHALL document the `useLegacyMicroBiomeThreshold` flag in JSDoc comments with a deprecation notice stating it will be removed in a future major version.
5. THE Test Suite SHALL include a test that verifies: WHEN `useLegacyMicroBiomeThreshold` is true, the system produces identical results to the current implementation for a fixed seed and configuration.

---

### Requirement 10: Documentation and Examples

**User Story:** As a library consumer, I want clear documentation explaining the micro-biome frequency parameter and terrain-aware placement, so that I can configure the system effectively.

#### Acceptance Criteria

1. THE EnhancedBiomeConfig interface SHALL include JSDoc comments for `microBiomeFrequency` explaining the valid range (0.0–0.5), the effect on density, and the interaction with terrain-aware placement.
2. THE EnhancedBiomeConfig interface SHALL include JSDoc comments for `depressionDepthThreshold` and `clearingGradientThreshold` explaining their effect on micro-biome placement and recommended value ranges.
3. THE examples directory SHALL include a new example file `micro-biome-tuning.ts` demonstrating how to configure micro-biome frequency and terrain thresholds for different aesthetic goals.
4. THE README.md file SHALL include a section on micro-biomes explaining the frequency parameter, terrain-aware placement, and visual feedback features.
5. THE example file SHALL include at least three configuration presets: "Sparse Micro-Biomes" (frequency 0.05), "Moderate Micro-Biomes" (frequency 0.15), and "Dense Micro-Biomes" (frequency 0.4).
