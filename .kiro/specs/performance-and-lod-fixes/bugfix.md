# Bugfix: Проблемы с производительностью и LOD системой

## Описание проблемы

Пользователь сообщает о двух критических проблемах:

1. **Недостаточная производительность**: Приложение не достигает стабильных 60 FPS
2. **Сломанная LOD система**: При включении LOD системы ломается ландшафт и в консоль выводятся ошибки

## Условие бага (Bug Condition)

Определим формальное условие бага C(X), где X - состояние системы:

### C1: Производительность < 60 FPS

```
C1(state) = state.fps < 60 AND state.loadedChunkCount > 0
```

**Ожидаемое поведение**: FPS должен быть >= 60 при любом количестве загруженных чанков (в разумных пределах)

**Фактическое поведение**: FPS падает ниже 60, особенно при большом количестве чанков

### C2: LOD система ломает ландшафт

```
C2(state) = state.lodManager != null AND (
  terrain_is_broken(state) OR 
  console_has_errors(state)
)
```

**Ожидаемое поведение**: 
- При включении LOD ландшафт должен отображаться корректно с разными уровнями детализации
- Никаких ошибок в консоли

**Фактическое поведение**:
- Ландшафт отображается некорректно (возможно, дыры, неправильная геометрия)
- В консоль выводятся ошибки

## Анализ корневых причин

### Проблема 1: Производительность

**Возможные причины**:

1. **Отсутствие оптимизаций рендеринга**
   - Frustum culling может быть отключен или работать неэффективно
   - Все чанки рендерятся каждый кадр, даже если они вне поля зрения

2. **Неэффективная генерация меши**
   - Создание геометрии может быть медленным
   - Отсутствие переиспользования объектов (object pooling)

3. **Слишком много вершин**
   - Высокое разрешение heightmap для всех чанков
   - LOD система не применяется или работает неправильно

4. **Проблемы с памятью**
   - Утечки памяти при создании/удалении чанков
   - Garbage collection паузы

### Проблема 2: LOD система

**Возможные причины**:

1. **Несоответствие размеров heightmap**
   - LOD система изменяет размер heightmap через `downsampleHeightmap`
   - Код создания меши ожидает размер `(chunkSize + 1) x (chunkSize + 1)` для seamless boundaries
   - После LOD downsampling размер может не совпадать

2. **Проблемы с индексацией**
   - В `createTerrainMesh` используется `verticesPerSide = chunkSize + 1`
   - Но после LOD downsampling `data.heightmap.length` может быть другим
   - Это приводит к неправильной индексации и ошибкам

3. **Отсутствие обновления размера чанка**
   - LOD система изменяет размер heightmap, но не обновляет `data.size`
   - Код рендеринга использует старый `data.size`, что приводит к несоответствию

## Проверка гипотез

### Гипотеза 1: LOD downsampling ломает размер heightmap

**Код в `lod.ts`**:
```typescript
private downsampleHeightmap(
  heightmap: Float32Array,
  size: number,
  resolution: number
): Float32Array {
  const newSize = Math.max(1, Math.floor(size * resolution));
  const newHeightmap = new Float32Array(newSize * newSize);
  // ...
  return newHeightmap;
}
```

**Код в `WorldViewer.ts`**:
```typescript
private createTerrainMesh(...) {
  const chunkSize = data.size; // Это СТАРЫЙ размер!
  const verticesPerSide = chunkSize + 1;
  const vertexCount = verticesPerSide * verticesPerSide;
  
  // Но heightmap имеет НОВЫЙ размер после LOD!
  const height = data.heightmap[index]; // ОШИБКА: index может быть вне границ!
}
```

**Вывод**: Это основная причина проблемы с LOD. После downsampling:
- `data.heightmap.length` = `newSize * newSize`
- Но `data.size` остается старым значением
- Код пытается обратиться к индексам, которых нет в новом heightmap

### Гипотеза 2: Frustum culling неэффективен

**Код в `WorldViewer.ts`**:
```typescript
private updateFrustumCulling(): void {
  // Проверяется каждые 100ms
  if (this.enableFrustumCulling && now - this.lastCullingCheck > this.cullingCheckInterval) {
    // ...
  }
}
```

**Проблема**: 100ms - это слишком редко для динамической камеры. При быстром движении камеры чанки могут оставаться видимыми, когда уже вне frustum.

### Гипотеза 3: Отсутствие LOD для дальних чанков

**Код в `DemoApp.ts`**:
```typescript
if (this.state.lodManager) {
  const lodLevel = this.getChunkLODLevel(chunkX, chunkY);
  chunk = this.state.lodManager.applyLOD(chunk, lodLevel);
  (chunk as any).lodLevel = lodLevel;
}
```

**Проблема**: LOD применяется, но если система сломана (гипотеза 1), то это только усугубляет проблему.

## Условие исправления (Fix Condition)

Исправление считается успешным, если:

### F1: Производительность восстановлена

```
F1(state) = state.fps >= 60 AND state.loadedChunkCount > 0
```

**Критерии**:
- FPS >= 60 при загрузке 9-25 чанков (3x3 - 5x5 сетка)
- FPS >= 45 при загрузке 49-100 чанков (7x7 - 10x10 сетка)
- Frustum culling работает корректно
- LOD система снижает нагрузку для дальних чанков

### F2: LOD система работает корректно

```
F2(state) = state.lodManager != null AND 
  terrain_is_correct(state) AND 
  !console_has_errors(state)
```

**Критерии**:
- Ландшафт отображается без дыр и артефактов
- Нет ошибок в консоли
- LOD уровни применяются корректно (HIGH/MEDIUM/LOW)
- Переходы между LOD уровнями плавные

## Стратегия исправления

### Исправление 1: Починить LOD систему

**Проблема**: Несоответствие размеров heightmap после downsampling

**Решение**:
1. Обновлять `data.size` после LOD downsampling
2. Или хранить оригинальный размер отдельно и использовать фактический размер heightmap
3. Или изменить `createTerrainMesh` для работы с любым размером heightmap

**Предпочтительное решение**: Обновлять `data.size` в `applyLOD`:

```typescript
applyLOD(chunk: any, level: LODLevel): any {
  const resolution = this.getMeshResolution(level);
  
  if (level === LODLevel.HIGH) {
    return chunk;
  }
  
  const newSize = Math.max(1, Math.floor(chunk.size * resolution));
  
  const lodChunk = {
    ...chunk,
    size: newSize, // ДОБАВИТЬ ЭТО!
    heightmap: this.downsampleHeightmap(chunk.heightmap, chunk.size, resolution),
    // ...
  };
  
  return lodChunk;
}
```

### Исправление 2: Оптимизировать frustum culling

**Проблема**: Проверка каждые 100ms слишком редкая

**Решение**:
1. Уменьшить интервал до 16ms (каждый кадр) или 33ms (каждый второй кадр)
2. Или использовать адаптивный интервал в зависимости от скорости движения камеры

### Исправление 3: Улучшить производительность рендеринга

**Решения**:
1. Убедиться, что frustum culling включен по умолчанию
2. Использовать object pooling для геометрии
3. Оптимизировать создание меши (уже есть в коде)
4. Добавить LOD для ресурсов и структур (не рендерить их на LOW LOD)

### Исправление 4: Добавить мониторинг производительности

**Решения**:
1. Добавить счетчик FPS в UI
2. Показывать количество видимых/скрытых чанков
3. Показывать LOD статистику

## Риски

1. **Изменение размера чанка может сломать другие части кода**
   - Митигация: Тщательное тестирование всех функций, использующих `data.size`

2. **Слишком агрессивный frustum culling может вызвать мерцание**
   - Митигация: Добавить небольшой margin для culling

3. **LOD переходы могут быть заметны**
   - Митигация: Использовать плавные переходы или увеличить расстояния между LOD уровнями
