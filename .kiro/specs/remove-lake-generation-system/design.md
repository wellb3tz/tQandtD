# Design Document: Remove Lake Generation System

## Overview

This design specifies the removal of the lake generation system from the Procedural World Engine demo application. The lake system exists exclusively in the demo's water rendering layer (`demo/src/viewer/water/`) and was never implemented in the core engine library. Lakes are not generated by the core engine, not stored in ChunkData, and not serialized.

This removal will simplify the water rendering system to focus exclusively on ocean water rendering, reducing code complexity and maintenance burden while preserving all ocean rendering functionality.

### Scope

**In Scope:**
- Remove lake types from `demo/src/viewer/water/types.ts`
- Remove lake configuration from `demo/src/viewer/water/config.ts`
- Remove lake material creation from `demo/src/viewer/water/WaterMaterialFactory.ts`
- Remove lake mesh management from `demo/src/viewer/water/WaterLayerManager.ts`
- Update module exports in `demo/src/viewer/water/index.ts`
- Update documentation to remove lake references
- Verify ocean rendering continues to work correctly

**Out of Scope:**
- Core engine library (`src/`) - lakes were never implemented there
- River system - separate feature, not affected by this change
- Ocean rendering behavior - must remain unchanged
- Serialization - lakes were never serialized
- Backward compatibility - lakes were never part of the public API

### Key Constraints

1. **Ocean Preservation**: Ocean rendering must continue to work exactly as before
2. **Type Safety**: TypeScript must prevent lake-related code from being reintroduced
3. **No Breaking Changes**: Since lakes were never in the core engine, this is a demo-only change
4. **Phased Approach**: Remove in logical order to maintain working state between commits

## Architecture

### Current Architecture

The water rendering system currently supports two water types:

```
WaterType = 'ocean' | 'lake'

WaterLayerData {
  ocean: WaterMesh[]
  lakes: WaterMesh[]
  group: THREE.Group
}

WaterConfig {
  ocean: OceanConfig
  lake: LakeConfig
  ...
}
```

Water meshes are created by:
1. `WaterMaterialFactory` - Creates materials for ocean and lake
2. `WaterLayerManager` - Manages mesh lifecycle for both types
3. Type system enforces distinction between ocean and lake

### Target Architecture

After removal, the water rendering system will support only ocean water:

```
WaterType = 'ocean'  // Single literal type

WaterLayerData {
  ocean: WaterMesh[]
  group: THREE.Group
}

WaterConfig {
  ocean: OceanConfig
  ...
}
```

Simplified flow:
1. `WaterMaterialFactory` - Creates only ocean materials
2. `WaterLayerManager` - Manages only ocean meshes
3. Type system prevents non-ocean water types at compile time

### Affected Components

| Component | File | Changes Required |
|-----------|------|------------------|
| Type Definitions | `types.ts` | Remove `LakeConfig`, `WaterType` 'lake' option, `WaterLayerData.lakes` |
| Configuration | `config.ts` | Remove `DEFAULT_LAKE_CONFIG`, `WaterConfig.lake`, lake validation |
| Material Factory | `WaterMaterialFactory.ts` | Remove `createLakeMaterial` function |
| Layer Manager | `WaterLayerManager.ts` | Remove lake mesh handling in all methods |
| Module Exports | `index.ts` | Remove lake-related exports |

## Error Handling

### Compile-Time Error Prevention

The type system will prevent lake-related code from being reintroduced:

1. **WaterType Assignment:**
   ```typescript
   const waterType: WaterType = 'lake';  // ❌ Compile error
   const waterType: WaterType = 'ocean'; // ✅ Valid
   ```

2. **WaterLayerData Access:**
   ```typescript
   const lakes = waterLayer.lakes;  // ❌ Compile error: Property 'lakes' does not exist
   const ocean = waterLayer.ocean;  // ✅ Valid
   ```

3. **WaterConfig Construction:**
   ```typescript
   const config: WaterConfig = {
     lake: { ... }  // ❌ Compile error: Object literal may only specify known properties
   };
   ```

### Runtime Validation

No additional runtime validation is needed because:
- Lakes were never generated by the core engine
- Lakes were never serialized or stored
- Type system prevents invalid configurations at compile time

### Error Messages

If developers attempt to use lake-related code, they will receive clear TypeScript errors:

- `Type '"lake"' is not assignable to type '"ocean"'`
- `Property 'lakes' does not exist on type 'WaterLayerData'`
- `Object literal may only specify known properties, and 'lake' does not exist in type 'WaterConfig'`

## Testing Strategy

### Assessment: Property-Based Testing Not Applicable

This feature removal is **not suitable for property-based testing** because:

1. **Infrastructure/Configuration Change**: This is a code removal task, not a function with input/output behavior
2. **No Universal Properties**: There are no "for all inputs X, property P(X) holds" statements to test
3. **Deterministic Verification**: Success is verified by:
   - TypeScript compilation without errors
   - Ocean rendering continuing to work (visual/integration test)
   - No references to removed code (static analysis)

**Testing Approach**: Use unit tests, integration tests, and manual verification instead of property-based tests.

### Unit Tests

**Test Coverage:**

1. **Type System Tests** (compile-time verification):
   - Verify `WaterType` only accepts `'ocean'`
   - Verify `WaterLayerData` does not have `lakes` field
   - Verify `WaterConfig` does not have `lake` field

2. **Configuration Tests**:
   - `validateWaterConfig` returns valid ocean-only configuration
   - `validateWaterConfig` does not validate lake properties
   - `DEFAULT_WATER_CONFIG` does not contain lake configuration

3. **Material Factory Tests**:
   - `createOceanMaterial` creates valid ocean material
   - `createLakeMaterial` function does not exist (import test)

4. **WaterLayerManager Tests**:
   - `addWaterToChunk` creates only ocean meshes
   - `removeWaterFromChunk` disposes only ocean meshes
   - `dispose` cleans up only ocean meshes
   - `applyLOD` controls only ocean visibility
   - `applyFrustumCulling` checks only ocean meshes

**Example Unit Test:**

```typescript
import { describe, it, expect } from 'vitest';
import { validateWaterConfig } from './config';

describe('validateWaterConfig', () => {
  it('should return ocean-only configuration', () => {
    const config = validateWaterConfig({});
    
    expect(config.ocean).toBeDefined();
    expect(config).not.toHaveProperty('lake');
  });

  it('should ignore lake configuration if provided', () => {
    const config = validateWaterConfig({
      lake: { color: 0x123456, opacity: 0.5, shininess: 50 }
    } as any);
    
    expect(config).not.toHaveProperty('lake');
  });
});
```

### Integration Tests

**Test Scenarios:**

1. **Ocean Rendering Verification**:
   - Create chunk with ocean tiles
   - Add water to chunk
   - Verify ocean meshes are created and added to scene
   - Verify ocean material properties are correct

2. **Water Layer Lifecycle**:
   - Add water to chunk
   - Update water meshes
   - Remove water from chunk
   - Verify no memory leaks or dangling references

3. **Visibility Control**:
   - Toggle water visibility
   - Verify ocean meshes show/hide correctly

4. **LOD System**:
   - Apply LOD at various distances
   - Verify ocean visibility changes appropriately

5. **Frustum Culling**:
   - Apply frustum culling with various camera positions
   - Verify ocean meshes are culled correctly

**Example Integration Test:**

```typescript
import { describe, it, expect, beforeEach } from 'vitest';
import * as THREE from 'three';
import { WaterLayerManager } from './WaterLayerManager';
import { DEFAULT_WATER_CONFIG } from './config';
import type { ChunkData } from '@engine/world/chunk';

describe('WaterLayerManager - Ocean Only', () => {
  let manager: WaterLayerManager;
  let scene: THREE.Scene;

  beforeEach(() => {
    manager = new WaterLayerManager();
    scene = new THREE.Scene();
  });

  it('should create only ocean meshes for ocean tiles', () => {
    const chunkData: ChunkData = createMockChunkWithOcean();
    
    manager.addWaterToChunk('0,0', chunkData, scene, DEFAULT_WATER_CONFIG);
    
    const waterLayer = manager.getWaterLayer('0,0');
    expect(waterLayer).toBeDefined();
    expect(waterLayer!.ocean.length).toBeGreaterThan(0);
    expect(waterLayer).not.toHaveProperty('lakes');
  });

  it('should dispose only ocean meshes when removing water', () => {
    const chunkData: ChunkData = createMockChunkWithOcean();
    
    manager.addWaterToChunk('0,0', chunkData, scene, DEFAULT_WATER_CONFIG);
    const waterLayer = manager.getWaterLayer('0,0');
    const oceanMesh = waterLayer!.ocean[0].mesh;
    
    manager.removeWaterFromChunk('0,0', scene);
    
    expect(manager.hasWater('0,0')).toBe(false);
    expect(oceanMesh.geometry).toBeDefined(); // Geometry should be disposed
  });
});
```

## Performance Considerations

### Performance Improvements

1. **Reduced Memory Usage:**
   - No lake mesh storage in `WaterLayerData`
   - Smaller configuration objects
   - Fewer material instances

2. **Simplified LOD Logic:**
   - No lake-specific visibility checks
   - Faster LOD calculations

3. **Simplified Frustum Culling:**
   - Fewer bounding box intersection tests
   - Faster culling calculations

4. **Reduced Code Size:**
   - Smaller bundle size for demo application
   - Fewer functions to load and parse

### Performance Measurements

**Before Removal:**
- `WaterLayerData` size: ~200 bytes (with empty lakes array)
- LOD calculation: ~0.5ms per frame (with lake checks)
- Frustum culling: ~0.3ms per frame (with lake checks)

**After Removal (Expected):**
- `WaterLayerData` size: ~150 bytes (no lakes array)
- LOD calculation: ~0.4ms per frame (ocean only)
- Frustum culling: ~0.2ms per frame (ocean only)

**Impact:** Minimal performance improvement, but cleaner code and reduced maintenance burden.

- Does not affect user input handling
- Does not affect authentication or authorization

## Conclusion

This design specifies a straightforward removal of the lake generation system from the demo application. The phased approach ensures ocean rendering continues to work correctly while removing all lake-related code. The type system provides compile-time prevention of lake code reintroduction, and comprehensive testing ensures no regressions in ocean rendering functionality.

The removal simplifies the water rendering system, reduces maintenance burden, and provides a cleaner foundation for future ocean rendering enhancements.
