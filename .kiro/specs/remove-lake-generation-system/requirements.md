# Requirements Document: Remove Lake Generation System

## Introduction

This document specifies the requirements for removing the lake generation system from the Procedural World Engine demo application. The lake system exists only in the demo's water rendering layer (demo/src/viewer/water/) and was never implemented in the core engine library. Lakes are not generated by the core engine, not stored in ChunkData, and not serialized. This removal will simplify the water rendering system to focus exclusively on ocean water rendering.

## Glossary

- **Lake_System**: The collection of types, configuration, materials, and mesh management code related to lake water rendering in the demo application
- **Water_Rendering_System**: The Three.js-based visualization system in demo/src/viewer/water/ that renders water features
- **Ocean_System**: The ocean water rendering components that must be preserved and continue functioning after lake removal
- **WaterLayerManager**: The class responsible for managing water mesh lifecycle (adding, removing, updating water meshes)
- **WaterMaterialFactory**: The factory module providing functions to create water materials
- **WaterConfig**: The configuration interface for the water rendering system
- **WaterLayerData**: The data structure containing water meshes for a chunk
- **Core_Engine**: The library code in src/ that generates world data (does not generate lakes)
- **Demo_Application**: The visualization application in demo/ that renders generated worlds

## Requirements

### Requirement 1: Remove Lake Types from Water System

**User Story:** As a developer maintaining the water rendering system, I want lake-related types removed from the type definitions, so that the type system only reflects implemented features.

#### Acceptance Criteria

1. THE Water_Rendering_System SHALL remove the LakeConfig interface from demo/src/viewer/water/types.ts
2. THE WaterConfig interface SHALL remove the lake configuration field
3. THE WaterLayerData interface SHALL remove the lakes field
4. THE WaterType type SHALL remove the 'lake' option, becoming a single literal type 'ocean'
5. FOR ALL water-related type definitions, lake references SHALL be absent

### Requirement 2: Remove Lake Configuration

**User Story:** As a demo application user, I want lake configuration options removed, so that I am not confused by non-functional configuration parameters.

#### Acceptance Criteria

1. THE Water_Rendering_System SHALL remove DEFAULT_LAKE_CONFIG from demo/src/viewer/water/config.ts
2. THE DEFAULT_WATER_CONFIG SHALL remove the lake configuration field
3. THE validateWaterConfig function SHALL remove lake configuration validation logic
4. WHEN creating water configuration, THE Configuration_System SHALL NOT accept lake-related parameters
5. FOR ALL water configuration objects, lake settings SHALL be absent

### Requirement 3: Remove Lake Material Creation

**User Story:** As a developer maintaining the material factory, I want the createLakeMaterial function removed, so that the factory only creates materials for implemented water types.

#### Acceptance Criteria

1. THE WaterMaterialFactory SHALL remove the createLakeMaterial function from demo/src/viewer/water/WaterMaterialFactory.ts
2. THE WaterMaterialFactory SHALL NOT import or reference LakeConfig
3. WHEN creating water materials, THE WaterMaterialFactory SHALL only provide ocean material creation
4. FOR ALL material factory exports, lake material creation SHALL be absent

### Requirement 4: Remove Lake Mesh Management from WaterLayerManager

**User Story:** As a developer using the water layer manager, I want lake mesh management removed, so that the manager focuses exclusively on ocean rendering.

#### Acceptance Criteria

1. THE WaterLayerManager SHALL remove lake mesh creation logic from the addWaterToChunk method in demo/src/viewer/water/WaterLayerManager.ts
2. THE WaterLayerManager SHALL remove lake mesh disposal logic from the removeWaterFromChunk method
3. THE WaterLayerManager SHALL remove lake mesh disposal logic from the dispose method
4. THE WaterLayerManager SHALL remove lake visibility logic from the applyLOD method
5. THE WaterLayerManager SHALL remove lake mesh checks from the applyFrustumCulling method
6. WHEN managing water meshes, THE WaterLayerManager SHALL only handle ocean meshes
7. FOR ALL water layer operations, lake mesh handling SHALL be absent

### Requirement 5: Update WaterLayerData Structure

**User Story:** As a developer working with water layer data, I want the data structure to only contain ocean meshes, so that the structure accurately represents the water system.

#### Acceptance Criteria

1. THE WaterLayerData interface SHALL contain only an ocean field (array of WaterMesh)
2. THE WaterLayerData interface SHALL contain a group field (THREE.Group container)
3. WHEN creating WaterLayerData instances, THE Water_Rendering_System SHALL NOT initialize a lakes array
4. FOR ALL WaterLayerData instances, the lakes field SHALL be absent

### Requirement 6: Simplify WaterType Type Definition

**User Story:** As a TypeScript developer, I want the WaterType to accurately reflect available water types, so that type checking prevents invalid water type usage.

#### Acceptance Criteria

1. THE WaterType type SHALL be defined as the literal type 'ocean' only
2. WHEN assigning WaterType values, THE Type_System SHALL only accept 'ocean'
3. THE Type_System SHALL reject 'lake' as an invalid WaterType value at compile time
4. FOR ALL WaterMesh instances, the type field SHALL only accept 'ocean'

### Requirement 7: Preserve Ocean Rendering Functionality

**User Story:** As a demo application user, I want ocean water rendering to continue working correctly after lake removal, so that I can visualize ocean water without interruption.

#### Acceptance Criteria

1. WHEN rendering chunks with ocean tiles, THE Water_Rendering_System SHALL create ocean water meshes
2. WHEN adding water to chunks, THE WaterLayerManager SHALL successfully add ocean meshes to the scene
3. WHEN removing water from chunks, THE WaterLayerManager SHALL successfully remove and dispose ocean meshes
4. WHEN updating water meshes, THE WaterLayerManager SHALL successfully regenerate ocean meshes
5. WHEN toggling water visibility, THE WaterLayerManager SHALL successfully show/hide ocean meshes
6. THE Ocean_System SHALL render with correct color, opacity, shininess, and positioning
7. FOR ALL ocean rendering operations, functionality SHALL remain unchanged from before lake removal

### Requirement 8: Update Water System Documentation

**User Story:** As a developer reading water system code, I want documentation to exclude lake references, so that I understand the actual capabilities of the water system.

#### Acceptance Criteria

1. THE Water_Rendering_System SHALL update JSDoc comments to remove lake references
2. THE WaterConfig interface documentation SHALL describe ocean-only configuration
3. THE WaterLayerManager documentation SHALL describe ocean-only mesh management
4. THE WaterMaterialFactory documentation SHALL describe ocean-only material creation
5. WHEN developers read inline documentation, THE Documentation SHALL NOT mention lake rendering capabilities

### Requirement 9: Remove Lake References from LOD System

**User Story:** As a developer using the LOD system, I want LOD logic to only handle ocean meshes, so that performance optimizations are simplified and accurate.

#### Acceptance Criteria

1. THE WaterLayerManager.applyLOD method SHALL remove lake-specific visibility logic
2. WHEN applying LOD at medium distance, THE LOD_System SHALL NOT attempt to hide lake meshes
3. THE LOD_System SHALL only control ocean mesh visibility based on distance
4. FOR ALL LOD operations, lake mesh handling SHALL be absent

### Requirement 10: Remove Lake References from Frustum Culling

**User Story:** As a developer using frustum culling, I want culling logic to only check ocean meshes, so that culling is efficient and accurate.

#### Acceptance Criteria

1. THE WaterLayerManager.applyFrustumCulling method SHALL remove lake mesh visibility checks
2. WHEN checking frustum intersection, THE Frustum_Culling_System SHALL only test ocean mesh bounding boxes
3. THE Frustum_Culling_System SHALL determine water layer visibility based solely on ocean meshes
4. FOR ALL frustum culling operations, lake mesh checks SHALL be absent

### Requirement 11: Verify No Broken References

**User Story:** As a library maintainer, I want to ensure no code references removed lake functionality, so that the codebase remains functional after removal.

#### Acceptance Criteria

1. THE Codebase SHALL NOT contain any imports of LakeConfig type
2. THE Codebase SHALL NOT contain any calls to createLakeMaterial function
3. THE Codebase SHALL NOT contain any references to waterLayer.lakes arrays
4. WHEN the TypeScript compiler runs, THE Build_System SHALL NOT report any errors related to missing lake types
5. WHEN tests run, THE Test_Suite SHALL NOT fail due to missing lake functionality
6. FOR ALL code files in demo/src/viewer/water/, lake-related imports and references SHALL be absent

### Requirement 12: Update Water System Tests

**User Story:** As a developer running tests, I want water system tests to exclude lake scenarios, so that tests accurately validate ocean-only functionality.

#### Acceptance Criteria

1. THE Test_Suite SHALL remove lake-related test cases from water system tests
2. THE Test_Suite SHALL remove lake configuration from test fixtures
3. THE Test_Suite SHALL remove lake mesh assertions from water layer tests
4. WHEN testing water rendering, THE Test_Suite SHALL only validate ocean rendering
5. FOR ALL water system tests, lake-related test code SHALL be absent

### Requirement 13: Maintain Type Safety After Removal

**User Story:** As a TypeScript developer, I want the type system to prevent lake-related code from being reintroduced, so that the codebase remains clean and consistent.

#### Acceptance Criteria

1. WHEN attempting to add lake configuration to WaterConfig, THE Type_System SHALL reject the code at compile time
2. WHEN attempting to assign 'lake' to WaterType, THE Type_System SHALL reject the code at compile time
3. WHEN attempting to access waterLayer.lakes, THE Type_System SHALL reject the code at compile time
4. THE Type_System SHALL enforce ocean-only water rendering through compile-time checks
5. FOR ALL water-related code, the type system SHALL prevent lake-related properties and values

### Requirement 14: Simplify Water Configuration Validation

**User Story:** As a developer using water configuration, I want validation logic simplified to only handle ocean configuration, so that configuration is easier to understand and maintain.

#### Acceptance Criteria

1. THE validateWaterConfig function SHALL only validate ocean configuration properties
2. THE validateWaterConfig function SHALL NOT include lake-specific validation logic
3. WHEN validating water configuration, THE Validation_System SHALL only check ocean color, opacity, shininess, wave settings
4. THE Validation_System SHALL NOT validate lake-specific properties
5. FOR ALL configuration validation, lake validation logic SHALL be absent

### Requirement 15: Update Water System Exports

**User Story:** As a developer importing water system modules, I want exports to exclude lake-related functionality, so that I only see available features in autocomplete.

#### Acceptance Criteria

1. THE WaterMaterialFactory module SHALL NOT export createLakeMaterial function
2. THE Water_System types module SHALL NOT export LakeConfig interface
3. WHEN importing from water system modules, THE Type_System SHALL NOT provide lake-related exports for autocomplete
4. FOR ALL water system module exports, lake-related functions and types SHALL be absent
